# Thryve Growth Co. — Project Guide

**What this is:** A career coaching platform for Rachel, an HR consultant and career coach. Clients book sessions, pay via Stripe, and access a personal dashboard with documents, job watchlists, and booking history. Rachel manages everything through an admin panel.

**Production URL:** `https://thryvegrowth.co`

---

## ⚠️ Critical: This Is Next.js 16

This project runs Next.js **16.2.2**, which has breaking changes from all prior versions:

| Breaking Change | Detail |
|---|---|
| `params` and `searchParams` are **Promises** | Always `await` them: `const { id } = await params` |
| `middleware.ts` is renamed | File is `src/proxy.ts`; exported function is `proxy()`, not `middleware()` |

**Before writing any Next.js-specific code**, read the relevant guide in `node_modules/next/dist/docs/`. Heed deprecation notices.

## Key Architectural Rules

**Header/Footer scope:** `Header` and `Footer` (`src/components/layout/`) render **only** inside `src/app/(marketing)/layout.tsx`. They are absent from dashboard, admin, and auth pages. The root `layout.tsx` is a bare HTML shell.

**Toast notifications:** `<Toaster />` (sonner) is in the root layout. Import `toast` from `"sonner"` in any client component.

**Route protection — two layers:**
1. `src/proxy.ts` — redirects unauthenticated users away from `/dashboard/*` and `/admin/*` to `/login?redirect=...`
2. Each admin page and server action checks `profiles.role = 'admin'` independently (proxy alone is not sufficient)

**Per-page admin check pattern:**
```typescript
const { data: profile } = await supabase.from("profiles").select("role").eq("id", user.id).single();
const p = profile as { role: string } | null;
if (p?.role !== "admin") redirect("/dashboard");
```
The `requireAdmin()` helper in `src/app/actions/blog.ts` and `src/app/actions/watchlist.ts` encapsulates this for server actions.

**Supabase — use the right client for the right context:**

| Context | File | Notes |
|---|---|---|
| Client components | `src/lib/supabase/client.ts` | Browser-safe |
| Server components / server actions / route handlers | `src/lib/supabase/server.ts` → `createClient()` | Uses cookies |
| Stripe webhook / admin server actions | `src/lib/supabase/service.ts` → `createServiceClient()` | Bypasses RLS — use only when caller is verified |

**Never use the service client in client components or in any unverified context.**

**TypeScript types (`src/types/database.ts`):**
- Hand-written (not auto-generated by Supabase CLI)
- Every table definition requires `Relationships: []` — without it, supabase-js v2 resolves result types to `never`
- Avoid `.eq()` on union-literal columns (e.g., `role`, `status`) in typed queries — the type narrows to `never`. Cast with `as LocalType[]` and filter in TypeScript, or annotate with `as any` + local type

**Stripe + Resend — lazy Proxy singletons:** Both clients are wrapped in a JavaScript `Proxy` to defer initialization until first access, preventing `next build` failures when secrets are absent.

**Blog content format:** Stored as Tiptap ProseMirror JSON (`JSONB`) in `blog_posts.content`. The extension sets in `RichTextEditor.tsx` (editor) and `blog/[slug]/page.tsx` (renderer) **must match** — mismatches cause empty output.

**Cron auth:** `GET /api/cron/job-alerts` requires `Authorization: Bearer {CRON_SECRET}`. In production Vercel sends this automatically. Absent locally, the endpoint allows all requests.

## Documentation Maintenance

Update the relevant doc file as part of every code change — not as an afterthought.

### Change → File Mapping

| Change type | Update this file |
|---|---|
| New/modified admin UI, screen, or workflow | `docs/rachel-admin-guide.md` |
| New route, component, server action, or API route | `docs/developer-architecture.md` |
| New/modified DB table, column, index, or RLS policy | `docs/database-schema.md` |
| Changes to a third-party integration or its config | `docs/integrations.md` |
| Env var added, removed, or renamed | `docs/environment-variables.md` |
| Key data flow change (booking, watchlist, blog, documents, auth, cron) | `docs/workflows.md` |

### Rules

1. Do not add fictional information — only document what the code actually does.
2. `docs/rachel-admin-guide.md` is for a non-technical admin — no code blocks, no TypeScript, plain language only.
3. All other docs are for developers — include file paths and function names.
4. If you add a Stripe price ID: update `docs/environment-variables.md` AND `src/lib/stripe/products.ts` — they must stay in sync.
5. If you rename or move a file referenced in docs, update the reference.
